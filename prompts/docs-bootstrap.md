# Docs Refresh — Bootstrap Mode (FIRST RUN ONLY)

You are generating `README.md` and `CLAUDE.md` from scratch for a repository that currently has none (or has trivially short ones). This is **high-risk** because there's no prior text to anchor against.

**This is a first-pass draft.** It will be pushed directly to the trigger
branch — there is no PR, no auto-reviewer, and no auto-merge gate. A human
must read the result and harden or revert it before relying on it.

## Inputs

- `/tmp/repo-context.json` — deterministic pre-scan: tech stack, package files, file tree, key entry points, line counts. **This is your primary source of facts.**
- The full repo working tree (read-only via Read/Glob/Grep)
- Recent git log (`git log --oneline -100`)
- Existing docs (if any): existing `README.md`, `CLAUDE.md`, `docs/` content — preserve any human-written sections verbatim

## Files you may create / modify

- `README.md` (create or augment, never replace existing prose)
- `CLAUDE.md` (create)
- `.claude/CLAUDE.md` (create only if `.claude/` directory already exists)

**You MUST NOT** modify any other file.

## Hard rules — read carefully

### Rule 1 — Evidence or `[VERIFY]`

Every concrete claim must be backed by one of:
- A file path you actually opened
- A grep result you actually ran
- A field you read from `/tmp/repo-context.json`

If you do not have evidence for a claim, you have **two options**:
- (a) Omit the claim entirely
- (b) Write `[VERIFY: <what to check>]` inline

**Never invent file paths, function names, env vars, ports, endpoints, command names, or version numbers.**

### Rule 2 — Show your work

In every doc you write, include a `## Sources` section at the bottom listing the files you read to write that doc. Example:

```
## Sources
This doc was generated from:
- package.json (scripts, dependencies)
- src/index.ts (entry point)
- src/routes/*.ts (endpoint inventory)
- docker-compose.yml (services)
```

A human reviewer will spot-check that what you wrote matches what those files actually contain.

### Rule 3 — Conservative scope

Write **short, dense, factual** docs. A 60-line CLAUDE.md that's all true beats a 400-line one that's half guessed.

Required sections in CLAUDE.md (only these — do not add others without evidence):
1. **One-paragraph description** of what the repo does
2. **Tech stack** (language, framework, key libraries — from package files)
3. **Build / test / dev commands** (from `package.json` scripts, `Makefile`, etc.)
4. **Project structure** (directory listing with one-line description per dir, only directories that exist)
5. **Key files** (entry points, with one-line description per file)
6. **Conventions** — only include if you can point to evidence (e.g., a `.eslintrc`, a `pyproject.toml [tool.ruff]`, an existing pattern visible across files)

Required sections in README.md:
1. **What this is** (one paragraph)
2. **Quick start** (install + run, taken verbatim from package scripts)
3. **Documentation pointers** (link to `CLAUDE.md`, `docs/` if present)

### Rule 4 — No invented features

Do not describe a feature unless you can point at the code that implements it. If you see `routes/auth.ts`, you can say "Has auth routes". You cannot say "Supports OAuth2 with PKCE" unless you actually grep for those terms and find them.

### Rule 5 — Banned content

- Do not write changelogs, "recent changes" lists, or commit history summaries.
- Do not write architecture diagrams (ASCII or otherwise) unless one already exists in code comments — they're high-hallucination-risk.
- Do not invent maintainer names, contact info, or links to non-existent docs.
- Do not add Co-Authored-By, Signed-off-by, or any AI authorship attribution.
- Do not include emojis unless the existing docs use them.

### Rule 6 — Marker prefix

Prepend this banner at the very top of any file you create or substantially expand:

```
> ⚠️ **AUTO-GENERATED FIRST DRAFT** — This document was generated by `.github/workflows/docs-refresh.yml` (bootstrap mode) and pushed directly to the branch without review. Every section needs human verification before relying on it. Search for `[VERIFY:` to find unconfirmed claims.
```

A human must remove this banner only after a full read-through.

## Workflow

1. Read `/tmp/repo-context.json` end to end.
2. Read every `README*`, `CLAUDE.md`, `.claude/CLAUDE.md`, and top-level `*.md` that already exists. Preserve them.
3. Read entry points and 5-10 representative source files.
4. List directories — if you can't open every file, say so and use file counts from the pre-scan.
5. Read `package.json` / `Cargo.toml` / `pyproject.toml` / `go.mod` for tech stack and scripts.
6. Write the docs. Be terse. Mark unknowns with `[VERIFY:]`.
7. Stop. Do not modify other files.

## When you're done

The workflow will stage tracked changes (`git add -u`), commit, and push
directly to the trigger branch. A human is expected to review the result and
either harden or revert it.

Be ruthlessly conservative. If you're unsure, write less.
